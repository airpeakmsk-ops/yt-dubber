# Plan 06-06 — Инкрементальное обновление леджера (design capture, 2026-07-06)

**Источник:** UAT Phase 6 (живой e2e) выявил дизайн-разрыв. Бот на леджере делает
FULL-REPLACE (`write_artifacts` пересобирает `prodazhi.parquet` целиком из присланного
файла). Рабочий процесс пользователя — ИНКРЕМЕНТАЛЬНЫЙ: присылает только новый период
ПОСЛЕ последнего учтённого; бот должен МЕРЖИТЬ, а не пересобирать.

Инцидент: база учтена до ~29.06.2026; пользователь прислал 01–06.07.2026 → бот затёр
всю историю 1-месячным куском (prodazhi 12656→314 строк, приход 122047→0, 33→1 мес).
Fail-safe не сработал (продаж >0). Данные восстановлены из бэкапа, Google откатил
пользователь.

## Подтверждённые решения пользователя (2026-07-06)

**Merge-семантика (по EAN, поверх распарсенных `parse_ledger` тоталов периода):**
- **Продажи** (`prodazhi.parquet`) → APPEND строк новых месяцев `(ean, month, qty)`.
- **Приход** (`prikhod_ledger.qty_prikhod`) → `+=` приход периода по EAN.
  Приход учитывает типы документов через `_doc_kind` (Поступление − Возврат
  поставщику; возвраты покупателя/перемещения различаются по «Документ движения
  (регистратор)» = `col1`). parse_ledger уже классифицирует — merge складывает
  per-EAN тоталы.
- **Остаток** (`prikhod_ledger.qty_stock`) → `новый = старый + дельта периода`
  (`Σ(приход−расход)` периода per EAN из `parse_ledger.ostatok_map`). Кумулятивно;
  дельта частичного файла (напр. −698) прибавляется к прежнему стоку.
- **EAN, отсутствующие в новом файле** → без изменений (база сохраняется).
- **Overlap-guard:** если ЛЮБОЙ месяц нового файла уже есть в `prodazhi.parquet` →
  ОТКЛОНИТЬ с предупреждением (защита от двойного счёта).

**Настройки выгрузки 1С для инкремента (пользователь):**
- Период = только новый диапазон (напр. 01.07.2026–06.07.2026), НЕ «не установлен».
- Группировки строк: Номенклатура + Номенклатура.Артикул (2 уровня; без «Документ
  движения»/«Регистратор» в группировках).
- Дополнительные поля: Период (→ дата в `col0`) + Документ движения (регистратор)
  (→ `col1`, нужен для `_doc_kind`: возвраты/перемещения).
- Показатели: Приход; Расход; Конечный остаток (Начальный НЕ нужен — работаем с дельтой).

## Открытые вопросы для плана
1. **Гранулярность overlap + шов частичного месяца (КРИТИЧНО).** `prodazhi` помесячный
   (строка на EAN за месяц). Два проявления одной проблемы:
   - Досылка частей месяца: 01–06.07 и позже 07–31.07 — оба «Июль 2026 г.» → overlap-reject
     заблокирует легитимную дозагрузку.
   - **Шов базы (реальный кейс 2026-07-06):** база учтена ДО 29.06.2026, т.е. «Июнь 2026 г.»
     в базе — ЧАСТИЧНЫЙ (до ~29.06). Пользователь верно заметил: инкремент надо слать
     С 30.06.2026, а не с 01.07 — иначе 30.06 теряется. Но «дошить» 30.06 = дополнить уже
     существующий месяц Июнь → month-level overlap-reject этому мешает.
   Варианты: (а) базовый cutoff выравнивать на границу месяца (перед переходом на инкремент
   догрузить полный июнь); (б) внутримесячная аккумуляция: `+=` qty к существующему месяцу
   с day-level дедупом (хранить обработанные диапазоны дат, а не только месяцы), overlap-reject
   применять по ДАТАМ, а не по месяцам. Рекомендация склоняется к (б): хранить «последнюю
   обработанную дату» и принимать только строки-движения строго ПОСЛЕ неё; месяц-строку
   prodazhi аккумулировать. Решить в плане — влияет на модель overlap-guard и стейт.
   NB: первый корректный инкремент должен стартовать с 30.06.2026 (день после последнего
   учтённого 29.06), не с 01.07.
2. **Кнопка/режим.** «Леджер» становится инкрементальным по умолчанию, или добавить
   отдельную «Леджер (полная замена)» для первичной/сброс-загрузки? Рекомендация:
   инкрементальный по умолчанию; full-replace — редкий явный режим.
3. **Хранение «последнего учтённого месяца»** для overlap — достаточно вывести
   `max(month)` из `prodazhi.parquet` (отдельный стейт не нужен).
4. Убрать/ослабить guard «приход==0 → reject» из `bot/pipeline._run_ledger`:
   для короткого инкремента приход=0 легитимен (за 6 дней могло не быть закупок).
   Заменить на инкремент-специфичные guard'ы (overlap, пустой период = 0 движений).
