## Phase Details

### Phase 1: Парсинг и сопоставление
**Goal**: Все исходные данные (приходы, продажи, остатки) разобраны и каждый товар имеет единый ключ, по которому можно агрегировать данные из трёх источников.
**Depends on**: Nothing (first phase)
**Requirements**: DATA-01, DATA-02, DATA-03, DATA-04, MATCH-01, MATCH-01b, MATCH-02
**Success Criteria** (что должно быть TRUE):
  1. Скрипт разбирает все файлы приходов (включая папку `в рублях/`) без ошибок через python-calamine и выдаёт датафрейм с колонками EAN, Товар, кол-во, цена руб, курс
  2. Скрипт разбирает `все продажи с 2023 по 26июня2026.xlsx` и выдаёт помесячные показатели (кол-во, стоимость, рентабельность) по каждой Номенклатуре без группировки Покупатель
  3. Скрипт разбирает `остатки все 260626.xlsx` и выдаёт суммарный остаток («Итог») по каждой Номенклатуре
  4. Для файлов без курса в имени курс получен из API ЦБ РФ по дате накладной и подставлен корректно
  5. Отчёт о несопоставленных позициях показывает, какие строки из продаж/остатков не нашли соответствия в приходах (и почему), процент покрытия >90%
**Plans**: 4 plans (3 waves)
  - [ ] 01-01-PLAN.md — Wave 0 pytest scaffold + foundation (normalize_ean, cbr_rates)
  - [ ] 01-02-PLAN.md — приход parser (21 files, курс+CBR) -> prikhody.parquet
  - [ ] 01-03-PLAN.md — продажи + остатки parsers -> prodazhi/ostatki.parquet
  - [ ] 01-04-PLAN.md — build_master join + unmatched coverage report

### Phase 2: Себестоимость USD
**Goal**: Каждый приход имеет себестоимость USD, и для каждой модели товара известна средневзвешенная себестоимость по всей истории закупок.
**Depends on**: Phase 1
**Requirements**: COST-01, COST-02, COST-03
**Success Criteria** (что должно быть TRUE):
  1. Для каждой строки прихода себестоимость USD вычислена по формуле `руб / курс / 1.038 / 1.16` и совпадает с ручным расчётом на 3 тестовых примерах
  2. Средневзвешенная себестоимость по модели учитывает разные курсы разных партий и корректно взвешена по количеству
  3. В итоговой структуре данных (до записи в Sheets) каждый приход несёт метаданные курса (число, источник: файл/ЦБ РФ)
**Plans**: 1 plan (1 wave)
  - [x] 02-01-PLAN.md — compute_cost: per-партия cost_usd + qty-weighted avg per EAN -> master_cost.parquet (COST-01/02/03)

### Phase 3: Основной отчёт в Google Sheets
**Goal**: В целевом Google Sheet появляется рабочий лист с полным единым списком товаров: приходы, помесячные продажи, накопительные итоги, скорость оборота и возраст остатка — без форматирования, но с корректными данными.
**Depends on**: Phase 1, Phase 2
**Requirements**: REPORT-01, REPORT-02, REPORT-03, REPORT-04, REPORT-05
**Success Criteria** (что должно быть TRUE):
  1. Лист «Отчёт» в целевом Google Sheet содержит все товары, которые когда-либо были в приходах (ни один товар не потерян при записи)
  2. Для каждого товара видны все приходы по датам с курсом и итоговые агрегаты, а рядом — помесячные продажи (кол-во) за каждый месяц от окт.2023 до текущего
  3. Колонки накопительного итога (кумулятив приходов и продаж) показывают нарастающие суммы, совпадающие с ручным подсчётом на 2 тестовых товарах
  4. Колонки DSI и шт/мес заполнены: DSI = текущий остаток / (суммарные продажи за период / кол-во месяцев), результат разумен (например, 45 дней для хорошо продающегося товара)
  5. Колонка возраст остатка (дней) показывает число дней с последнего прихода до сегодня для каждого товара
**Plans**: 2 plans (2 waves)
  - [ ] 03-01-PLAN.md — сборка отчёта в pandas (метрики + build_report_df, REPORT-01..05, офлайн-тесты)
  - [ ] 03-02-PLAN.md — запись в Google Sheets (idempotent, creds из env, .gitignore, мок-тест + human-gate live)

### Phase 4: Ранжирование, сезонность и план дозаказа
**Goal**: Основной лист окрашен по правилам покрытия и % продаж, отсортирован «горящие сверху», сезонные индексы посчитаны, и для каждого подходящего товара есть скорректированный план дозаказа на 2 месяца.
**Depends on**: Phase 3
**Requirements**: VISUAL-01, VISUAL-02, VISUAL-03, VISUAL-04, ORDER-01, ORDER-02, SEASON-01, SEASON-02
**Success Criteria** (что должно быть TRUE):
  1. Строки с DSI < 30 дней (менее 1 мес) окрашены красным, 30–60 жёлтым, 60–90 зелёным, >90 синим; сортировка выставлена красные сначала — проверяемо визуально на живой таблице
  2. Колонка «% продаж к приходам» заполнена и ячейки окрашены по 5-уровневой шкале (<20% красный … 80–100% зелёный)
  3. Сезонные индексы по 12 месяцам рассчитаны из исторических продаж и пики март–май и сент–ноябрь имеют индексы >1 (проверяется на листе или в данных) [known deviation: ноябрь=0.78<1 из-за скудных данных ноября 2023 — это не ошибка, пики сент/окт подтверждены]
  4. Товары с нулевыми продажами за последние 12 месяцев помечены как «мёртвый остаток» в отдельной колонке
  5. Для товаров с долей продаж ≥60% колонка «к заказу на 2 мес» содержит положительное число (или 0, если остаток уже покрывает), скорректированное на сезонный индекс ближайших 2 месяцев
**Plans**: 4 plans (3 waves)
  - [ ] 04-01-PLAN.md — Wave 0 pytest scaffold + парсер недельных остатков + availability velocity/DSI (контрактный sweep)
  - [ ] 04-02-PLAN.md — сезонные индексы (global + per-model через model_aliases, fallback)
  - [ ] 04-03-PLAN.md — order_plan: % продаж, к заказу (порог 60% + сезон), мёртвый/залежалый, presort
  - [ ] 04-04-PLAN.md — build_report 84 cols + apply_formatting (batch_format) + лист «Сезонность» + live write (human-verify)

### Phase 5: Дашборды
**Goal**: В Google Sheet появляются 4 листа-дашборда, каждый из которых самостоятельно отвечает на один вопрос без необходимости листать основной отчёт.
**Depends on**: Phase 4
**Requirements**: DASH-01, DASH-02, DASH-03, DASH-04
**Success Criteria** (что должно быть TRUE):
  1. Лист «Сезонность» показывает диаграмму или sparklines помесячных продаж по категориям/моделям с видимыми пиками сезона
  2. Лист «Топ-оборот» показывает топ-20 самых быстро продающихся товаров, отсортированных по шт/мес
  3. Лист «Залежавшиеся» показывает товары, отсортированные по убыванию возраста остатка (дней), с колонкой стоимости мёртвого капитала
  4. Лист «План дозаказа» показывает итоговую таблицу: товар, текущий остаток, оборот/мес, к заказу, сезонный индекс — отсортированную по приоритету (красные сверху)

### Phase 6: Telegram-бот обновления
**Goal**: Пользователь может с телефона прислать в Telegram свежий файл складских данных (единый леджер «приходы остатки», недельные остатки или накладную), выбрать его тип кнопкой, и через минуту целевой Google Sheet обновлён актуальными данными с подтверждением «обновлено N строк».
**Depends on**: Phase 1, Phase 2, Phase 3, Phase 4, Phase 5
**Requirements**: BOT-01, BOT-02, BOT-03, BOT-04
**Success Criteria** (что должно быть TRUE):
  1. Бот принимает xlsx-файл в Telegram, после выбора типа кнопкой запускает пересчёт и возвращает подтверждение с числом обновлённых строк листа «Отчёт»
  2. [контракт обновлён под единый леджер] Если прислан только недельный файл — пересчитывается только отчёт (parquet приходов/себестоимости не затираются)
  3. [контракт обновлён под единый леджер] Если прислан леджер — полный пересчёт (приход+продажи+себестоимость+отчёт); если накладная — пересчёт себестоимости и отчёта
  4. Бот отправляет текст-напоминание об обновлении еженедельно (понедельник 09:00 МСК) единственному whitelisted-пользователю
**Plans**: 5 plans (4 waves)
  - [x] 06-01-PLAN.md — Wave 0 scaffold + контракт main()->int (sweep) + config/keyboards (completed 2026-06-30)
  - [ ] 06-02-PLAN.md — backup/restore/validate (бэкап+откат+валидация, fail-closed)
  - [ ] 06-03-PLAN.md — pipeline orchestrator (карта тип→шаги: ledger/weekly/invoice)
  - [ ] 06-04-PLAN.md — handlers + scheduler + main (BOT-01 приём, BOT-04 пинг, whitelist)
  - [ ] 06-05-PLAN.md — деплой на VPS (systemd, secrets, live e2e — checkpoints)
